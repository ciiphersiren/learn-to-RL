HOW RL WORKS — CODE WALKTHROUGH

CHAPTER 1 / 6 · THE BIG PICTURE

What even IS
reinforcement learning?

Forget the math for a second. RL is literally just this: an agent tries stuff, gets punished or rewarded, and gradually learns what to do. Same way a dog learns tricks.

The 3 things every RL system needs An environment (the world), an agent (the learner), and a reward signal (feedback). Everything else is math on top of this.

In the playground, the environment is the grid. The agent is the yellow circle. The reward is +10 for the goal, −5 for traps, −0.05 per step.

The agent starts knowing absolutely nothing. It doesn’t know where the goal is. It doesn’t know walls exist. It just… tries things. And slowly, through thousands of tiny updates, it learns.

Why not just hardcode the path? You could — but it only works for THIS grid. RL agents learn a general policy that works even when the environment changes. That’s the superpower.

The algorithm used is called Q-Learning. “Q” stands for “quality” — it learns how good each action is in each state.

CHAPTER 2 / 6 · THE ENVIRONMENT

Building the world
the agent lives in

Every RL system needs a world. In code it’s just a data structure that describes states, allows actions, and returns rewards.

Grid cell values 0 = walkable  |  1 = wall  |  2 = goal (+10)  |  3 = trap (−5)

The agent’s state is just its position: row and column. That’s the entire state space. It can take 4 actions: up, down, left, right, encoded as [Δrow, Δcol].

What happens each step Pick action → compute new position → wall? stay put (−0.5) → goal? +10, done → trap? −5, done → otherwise −0.05, continue.

The reward function is the most critical design decision in RL. The −0.05 step penalty makes the agent want to reach the goal fast, not just eventually.

CHAPTER 3 / 6 · THE BRAIN

The Q-Table:
the agent’s memory

We need the agent to remember “in state S, how good is action A?” We store this in a table called the Q-table.

Q-Table structure A dictionary. Key = state string (e.g. “3,4”). Value = array of 4 numbers [up, down, left, right]. Each number = expected total future reward from taking that action.

At the start, everything is 0. As it explores, these numbers get updated to reflect reality.

// Q-table after some learning:
Q[“3,4”] = [-0.2, 4.8, -1.1, 3.2]
          ↑up ↓down ←left →right
// Best action = DOWN (4.8 is highest)

The arrows on the grid are literally the Q-table. Loop every cell, find the highest Q-value, draw the matching arrow. That’s the whole visualization.

Key insight The Q-table is NOT a neural network. It’s a plain JavaScript object. Q-learning is elegant because it’s just a fancy way to update a lookup table.
CHAPTER 4 / 6 · THE MATH

The Q-Learning
update rule

After every single step, we run this formula. This one line is what makes learning happen.

Q(s,a) ← Q(s,a) + α × [r + γ × maxQ(s′) − Q(s,a)]

s  = current state     r  = reward received
α = learning rate (0.2)
γ = discount factor (0.95)
maxQ(s′) = best Q in the NEXT state

The part in brackets is the TD error — how wrong was our previous estimate? We then nudge the old estimate a small step toward the truth.

Intuition for humans You think a path is worth 3 pts. You step, get −0.05, and the best forward path looks worth 4 pts. Reality = −0.05 + 0.95×4 = 3.75. You were too low! Move your estimate up toward 3.75. That’s ALL Q-learning is.

Learning rate α = 0.2: move only 20% toward the new estimate. Too high → unstable. Too low → learns forever.

Discount γ = 0.95: rewards 10 steps away are worth 0.95¹° ≈ 60%. Care about the future, but not infinitely.

CHAPTER 5 / 6 · EXPLORATION

The explore vs exploit
dilemma

If the agent only does what it currently thinks is best, it’ll miss better paths it never tried. But if it always explores randomly, it never uses what it learned.

ε-greedy rule With probability ε, take a random action (explore). With probability 1−ε, take the best known action (exploit). Start ε=1.0, decay slowly to 0.05.

This is why in the playground the agent starts completely random and gradually improves — ε decays from 1.0 to 0.05 over episodes, shifting from pure exploration to mostly exploitation.

The decay formula After each episode: epsilon = max(0.05, epsilon * 0.992)
Slowly shrinks to 0.05 and stays there — the agent never fully stops exploring.
Try it in the playground! Episodes 0–30 look chaotic (high ε). By ep 100 paths look purposeful. By ep 200+ it’s nearly optimal. The visual difference IS epsilon decay.
CHAPTER 6 / 6 · THE VISUALS

How the animation
actually works

The simulation and visuals are completely separate. The RL code just updates numbers. A rendering loop draws them.

The HTML Canvas API Two layered <canvas> elements. Every frame we clear them and redraw everything from scratch using the current Q-table and agent position.

The game loop is setInterval(tick, delay). Every tick: run N RL steps → call render() → draw everything fresh. That’s the animation.

The Q-value arrows: loop every cell, read Q-values, find the max, draw that arrow character. The Q-table IS the arrows. No magic.

The live charts: every episode push a value to an array. Redraw the chart canvas by mapping each array value to a pixel coordinate and drawing a line through all of them.

The key insight about “real-time” No streaming. No websockets. Just: run RL step → push to array → redraw chart. Everything in one browser tab. The live feeling is just drawing very fast.
The full stack HTML + CSS + vanilla JavaScript. No frameworks, no libraries, no backend. The entire RL agent + visualization is ~200 lines of JS.
